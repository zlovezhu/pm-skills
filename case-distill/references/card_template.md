# Card 标准格式

> 给 `case-distill` Skill 用。从 wiki 中抽离出来的"一句话原则"按这个格式写。

## 一、Frontmatter 规范

```yaml
---
type: card
date: 2026-05-19
tags: [{原则适用的领域 tag}]
source_wiki: 2026-05-19-排查-{slug}.md
applicable_to:
  - 设计交互响应阈值时
  - 评审取消/撤销类操作时
status: 已验证   # 或 经验性 / 待验证
---
```

`status` 字段：
- **已验证**：在多个 case 里都成立
- **经验性**：来自单次 case，未广泛验证
- **待验证**：直觉判断，没充分依据

## 二、Card 骨架

```markdown
# {一句话原则陈述句}

> {再用 1~2 句话展开，说清楚为什么这条原则成立}

## 适用场景

- {场景 1}
- {场景 2}

## 反例 / 边界

{这条原则不适用的情况，避免被泛用}

## 来源

来自 [[{source_wiki}]] 中的 {第几段}。
```

> 注意：card 全文应控制在 200 字以内（不含 frontmatter）。
> 简化口诀：**能独立引用的"一句话原则" = Card；带过程和现场的"一份方案" = Wiki。**

## 三、完整示例

```markdown
---
type: card
date: 2026-05-19
tags: [交互设计, B端工具]
source_wiki: 2026-05-19-排查-{slug}.md
applicable_to:
  - 设计交互响应阈值时
  - 评审取消/撤销类操作时
status: 经验性
---

# 取消类操作的视觉反馈应在 200ms 内完成

> 用户点击"取消 / 撤销 / 关闭"时，潜意识在等"什么都不再发生"的确认信号。
> 超过 200ms 没有视觉变化，会被判定为"按钮坏了"，再点第二次就成了灾难性误操作。

## 适用场景

- 操作范围/选区的取消按钮
- 弹窗关闭
- 任何"撤销当前临时状态"的交互

## 反例 / 边界

- 涉及网络请求的"取消订单"这类操作不适用——用户能接受 loading
- 复杂动画过渡可以超过 200ms，但要在 200ms 内**先开始**变化

## 来源

来自 [[2026-05-19-排查-{slug}]] 中的"5. 落到代码 / 行动"段。
```

## 四、抽 card 的反面例子

下面这些不应该抽成 card（应该留在 wiki 里）：

❌ **「{某模块} {某模式} 状态机里要补 cancel 转移」**
   理由：依附特定模块特定文件，无法独立引用。

❌ **「我们用 React + Zustand 管理状态」**
   理由：是事实陈述，不是原则。

❌ **「这个 bug 是因为状态机没处理 cancel 事件」**
   理由：是 case 的根因，不是可复用结论。

正确抽出的 card 例子：

✅ **「取消类操作的视觉反馈应在 200ms 内完成」**
✅ **「异步解析的首屏阈值参考：建索引 ≤ 10s，命中查询 ≤ 1s」**
✅ **「B 端专业工具的核心价值是缩短从发现到定位的路径」**

共同点：都是**脱离具体 case 仍然成立的判断**。
