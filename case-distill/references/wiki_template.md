# Wiki 五段式模板

> 给 `case-distill` Skill 用。所有 wiki 笔记按这个结构写。

## 一、Frontmatter 规范

每篇 wiki 顶部都加：

```yaml
---
type: wiki
date: 2026-05-19
module: {模块名}        # 按本项目实际模块结构填写
tags: [{模块}, {子特性}, 排查]
related_raw: 2026-05-19-排查-{slug}-raw.md
related_cards:
  - {可复用原则 1}
  - {可复用原则 2}
status: 已收敛   # 或 待验证 / 待补充
---
```

`status` 字段说明：
- **已收敛**：根因明确、行动闭环
- **待验证**：方向收敛但还没研发实测
- **待补充**：缺关键证据，需要后续找人对齐

## 二、五段式骨架

```markdown
# {陈述句标题，能脱离文件名独立读懂}

{触发这次排查/讨论的现象}

- **来源**：[谁 / 哪个渠道] 反馈
- **初始报错或反馈**：{原话或现象描述}

## 2. 看现场

{当时观察到的关键证据}
- 截图：![](path/to/screenshot.png)
- 日志片段：
  ```
  [HH:MM:SS] ERROR ...
  ```
- 关键指标：{数值 / 趋势}

## 3. 还原过程

{排查/讨论的关键节点，按时间线}

| 时间 | 节点 | 试了什么 | 看到了什么 | 由此判断 |
|---|---|---|---|---|
| HH:MM | {节点1} | {动作} | {观察} | {推断} |
| HH:MM | {节点2} | ... | ... | ... |

或者用列表式（节点少时）：

1. **HH:MM 第一个怀疑方向：{X}**
   - 试了：{动作}
   - 看到：{观察}
   - 由此判断：{推断}（或"否决，因为……"）

2. **HH:MM 第二个怀疑方向：{Y}**
   - ...

## 4. 找热点

{最终定位到的根因 / 关键约束}

- **根因**：{一句话说清楚}
- **支撑证据**：
  - {证据 1}
  - {证据 2}
- **影响范围**：{哪些场景 / 用户 / 模块会被波及}

## 5. 落到代码 / 行动

{解决方案 + 后续 action}

- **解决方案**：{技术方向 / 产品调整}
- **行动项**：
  - [ ] {action 1}（负责人 / DDL）
  - [ ] {action 2}
- **对产品/平台的影响**：{需要更新的功能、需要新增的兜底、需要补的文档}
```

## 三、完整示例

```markdown
---
type: wiki
date: 2026-05-19
module: {模块 A}
tags: [{模块 A}, {子特性}, 取消操作, 排查]
related_raw: 2026-05-19-排查-{模块A}{子特性}取消按钮失效-raw.md
related_cards:
  - 取消交互的响应阈值
status: 待验证
---

# {模块 A} {子特性}下取消按钮失效的排查

## 1. 发现异常

- **时间**：2026-05-19 14:23
- **来源**：研发 {同事昵称} 在团队聊天工具反馈
- **初始报错或反馈**：「{子特性}模式下操作之后取消按钮点了没反应」

## 2. 看现场

- 日志片段（main-20260519-142311.log）：
  ```
  [14:23:11] ERROR [{ModuleA}{Feature}] cancel invoked but state is INVALID
  [14:23:15] ERROR [{ModuleA}{Feature}] operation still active after cancel
  ```
- 复现路径：打开 {子特性} 模式 → 触发操作 → 点取消 → 视觉无变化

## 3. 还原过程

1. **14:23 第一个怀疑方向：按钮事件未绑定**
   - 试了：检查 React DevTools，按钮 onClick 有触发
   - 由此判断：事件层正常，否决该方向

2. **14:30 第二个怀疑方向：状态机未处理 cancel 事件**
   - 试了：看日志 `state is INVALID` 这条
   - 看到：cancel 事件触发时，状态机还停留在"operation-pending"
   - 由此判断：cancel 没被状态机正确接收

3. **14:50 收敛**
   - 根因落在 {子特性} 模式的状态机里没注册 cancel 转移

## 4. 找热点

- **根因**：{子特性} 模式状态机缺 `operation-pending → idle` 的转移
- **支撑证据**：源码中 `{moduleA}{Feature}Machine.ts` 的 transitions 表里没有 cancel
- **影响范围**：所有 {子特性} 模式下的取消操作；非 {子特性} 模式不受影响

## 5. 落到代码 / 行动

- **解决方案**：补一条状态转移，cancel 时强制回到 idle 并清空当前操作状态
- **行动项**：
  - [ ] {同事昵称} 改状态机（DDL: 5/22）
  - [ ] 我加一条 PRD 验收点：取消按钮在 200ms 内必须有视觉反馈
- **对产品/平台的影响**：
  - PRD 「{模块 A} 支持 {子特性} 操作」需要补充取消交互的验收标准
```
