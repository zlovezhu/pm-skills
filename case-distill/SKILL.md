# Case Distill —— 把一次 case 沉淀成可复用的知识

## 这个 Skill 解决什么

在真实的产品工作里，每天会经历的事：
- 和研发讨论一个性能问题，最终想到了某个解决方向
- 自己花半天排查一个奇怪的 bug，找到了根因
- 做完一个产品方案的内部评审，收敛到了一份决策

这些"过程"如果不及时沉淀，过几天就只剩"我好像处理过这事"。Obsidian 知识库的三层流转（raw → wiki → card）就是为了解决这个问题，但每次手动整理成本高。

这个 Skill 的职责：**接收任意形态的 case 原始输入，按"先发现异常 → 看现场 → 还原过程 → 找热点 → 落到代码"的叙事顺序整理成 wiki，并从中抽离 0~3 张 card；同时保留一份 raw 副本，避免一次产出错过原始信息。**

## 适用场景

- 把一段 IM 聊天里和研发的讨论沉淀
- 把自己的排查散记整理成一份可读 wiki
- 把已写好的复盘文档卡片化（抽出可复用原则）
- 多个来源混合（聊天 + 自己批注 + 文档），统一沉淀

## 核心规则

### 规则 1：默认落 wiki，但同时保留 raw 副本

每次 distill 输出三份内容：

```
1. {date}-{slug}.md            → 写到 1-Projects/<YOUR_PROJECT>/  （或对应模块的 wiki 目录）
2. {date}-{slug}-raw.md        → 写到 0-Inbox/raw/                （原始素材副本，便于后续二次提炼）
3. cards/{原则关键词}.md       → 写到 Cards/                      （0~3 张，按规则 3 判断）
```

raw 副本的存在不是冗余——它是"未消化的原始上下文"，等几个月后想验证 wiki 结论是否还成立时可以回查。

### 规则 2：wiki 叙事顺序固定

所有 wiki 笔记按下面五段式组织：

```markdown
## 1. 发现异常
{触发这次排查/讨论的现象，含时间、来源、初看到的报错或反馈}

## 2. 看现场
{当时观察到的关键证据：截图、日志片段、数据指标}

## 3. 还原过程
{排查/讨论的关键节点，按时间线，每个节点说"试了什么 / 看到了什么 / 由此判断"}

## 4. 找热点
{最终定位到的根因或关键约束}

## 5. 落到代码 / 行动
{解决方案、后续 action item、对产品/平台的影响}
```

如果某段确实没素材，留 `_（暂无信息）_` 占位，不要硬编。

### 规则 3：Card 抽取的判断标准

从 wiki 中抽 card 的三问（命中即抽）：

> **问 1：这条结论能不能脱离当前 case 单独引用？**
>   能 → Card 候选
>   不能（必须依附具体项目/模块/数据）→ 留在 wiki 里
>
> **问 2：1~3 句话能讲完，没有步骤、没有截图依赖？**
>   能 → Card 候选
>   不能（有步骤、有现场、有"前因后果"叙事）→ 留在 wiki 里
>
> **问 3：未来希望它作为"原则提示"自动跳出来？**
>   是 → Card 候选
>   否（只是想"翻出来照着做一遍"）→ 留在 wiki 里

**简化口诀（写进每张 card 的 frontmatter 注释）**：

> 能独立引用的"一句话原则" = Card；带过程和现场的"一份方案" = Wiki。

每次 distill 的标准产出 = **1 篇 wiki + 0~3 张 card**。

零 card 是允许的——不是每次 case 都能抽出原则，强抽反而稀释 card 的信号密度。

### 规则 4：标题与命名

- **wiki 文件名**：`{YYYY-MM-DD}-{核心动作}-{核心对象}.md`
  - 示例：`2026-05-19-排查-列表批量操作按钮失效.md`
- **wiki 内 H1 标题**：陈述句，能脱离文件名独立读懂
  - 示例：`列表筛选模式下批量操作按钮失效的排查`
- **card 文件名**：`{原则关键词}.md`，原则关键词 ≤ 12 字
  - 示例：`异步解析的可接受首屏阈值.md`
- **card 内 H1**：直接就是那条原则的一句话陈述
  - 示例：`异步解析 + 首次建索引可接受 10s 量级，关键看后续匹配是否在 1s 内`

### 规则 5：避免 AI 味

每次 distill 的最终输出在写出前要按"去AI味"的检查项过一遍（建议配套使用同名 Skill）：
- 不用"路径-信息-证据"这类抽象框架词，用"先发现异常 → 看现场 → 还原过程"这种贴近真实分析过程的措辞
- 不用三段并列的"鼓掌句"
- 不用"赋能 / 抓手 / 闭环 / 链路 / 范式"这类词，除非它就是项目里实际的术语
- 不用过度的形容词堆砌

## 工作流程

### Step 1：识别输入形态

判断输入是哪几类（可同时存在多种）：
- 聊天记录（IM 截图 / 文字粘贴）
- 散记（自己的排查笔记 / 想法）
- 已写好的复盘文档
- 多源混合

### Step 2：先做一份 raw 副本

不管输入是什么，先**原样**保存一份到 `0-Inbox/raw/{date}-{slug}-raw.md`。
这份不做任何加工，只在头部加一段 frontmatter：

```yaml
---
type: raw
date: YYYY-MM-DD
source: [IM 聊天 + 个人散记]
related_wiki: YYYY-MM-DD-排查-列表批量操作按钮失效.md
---
```

### Step 3：按五段式起草 wiki

按规则 2 的五段式组织，每段从 raw 里挑素材填充。

如果某段实在没素材，明确写 `_（暂无信息）_`，并在 wiki 末尾的"待补充"区列出来。

### Step 4：抽 Card 候选

从 wiki 草稿里扫一遍，按规则 3 的三问判断。
每张候选 card 用一句话写出来，列给用户：

```
我从这次 case 里识别到 {N} 张可能的 card 候选：
1. [建议抽] 异步解析 + 首次建索引可接受 10s 量级，关键看后续匹配是否在 1s 内
   理由：能独立引用、一句话讲完、可作为后续做异步功能时的原则提示
2. [建议留 wiki] 多端策略不该一刀切
   理由：脱离多端具体场景就讲不清，更适合留在 wiki 叙事里
   ...
```

让用户最终拍板抽哪些。

### Step 5：写入 Obsidian + 回执

用户确认后：
1. 写入 wiki 到对应模块目录（按规则 4 命名）
2. 写入 raw 副本到 `0-Inbox/raw/`
3. 写入选中的 card 到 `Cards/`
4. 给用户一份回执（见下面"回执模板"）

## 回执模板

```
🪶 已沉淀 · {wiki 标题}
─────────────────────────
📄 wiki：[{文件名}]({Obsidian 链接})
📁 raw：[{文件名}]({Obsidian 链接})
🃏 cards：
  · [{原则}]({Obsidian 链接})
  · [{原则}]({Obsidian 链接})

📌 待补充：
  · {wiki 中标 _（暂无信息）_ 的段落}

下一步建议：
  · {如果 wiki 涉及未确认的方向，提示去验证}
  · {如果某段证据缺失，建议下次找研发对齐时顺手补上}
```

## 不做什么（边界）

- 不**自动**写入 Obsidian——必须等用户在 Step 4 确认 card 候选之后再写
- 不**修改**已有的 wiki——如果检测到同主题已有 wiki，要先告诉用户"已存在 X，是覆盖、追加还是重新建一篇"
- 不**抽超过 3 张 card**——card 信号密度比数量更重要，宁缺毋滥
- 不**跨项目猜测**——只基于本次 case 的素材沉淀，不引入"我以为相关"的外部内容

## 与其他 Skill 的协作

- **上游**：`tapd-requirement-record` 落完缺陷单后，如果是个值得复盘的 case，可以接本 Skill
- **横向**：`prd-context-fetch` 检索时如果发现某模块"还没沉淀 wiki"，可以提示用户用本 Skill 补上
- **风格**：每次输出前用 `去AI味` Skill 过一遍

## 配置说明

使用前需要把以下占位符替换成你的实际值：

- `<YOUR_PROJECT>`：你的项目名，对应 Obsidian vault 中 `1-Projects/` 下的子目录
- 默认 Obsidian vault 采用 PARA 结构（0-Inbox / 1-Projects / Cards），如不一致请调整规则 1 的写入路径
