# TAPD Requirement Record —— 一句话需求/缺陷快速结构化

## 这个 Skill 解决什么

真实的产品工作里，需求和缺陷几乎从不是从模板开始写的，而是：
- IM 群里飘过来一句「这个能不能加个 XX」
- 周会上随口提了一个想法，自己手记一行
- 研发甩了一份日志文件，让你"提个单跟一下"
- 自己排查时发现的小问题，散记在某个文档里

这些"半成品输入"要落到 TAPD，痛点是：
1. 字段一堆，每次都要照着模板对一遍
2. 一句话需求 / 一行报错，扩成完整的"背景—详情—预期效果" / "问题描述—重现步骤—实际/预期"很费力
3. 日志一大段，到底要不要全粘进去、什么算关键报错？

这个 Skill 的职责是：**把任何"半成品输入"接进来，按 TAPD 字段模板补完，缺什么必填项才反问，能推断的全部直接填好；最终给一段可直接走 TAPD MCP 创建的结构化数据 + 一份给用户看的简洁回执。**

## 适用场景

- IM 聊天截图 / 文字粘贴 → 提 TAPD 需求或缺陷
- 会议纪要里的零散需求点 → 拆成多条 TAPD 单
- 自己的散记 / 语音转文字 → 整理成一条规范单
- 研发甩过来的日志文件 → 抽关键信息建缺陷单
- 多种输入混合（截图 + 日志 + 一段对话）

## 核心规则

### 规则 1：先判断单类型（需求 vs 缺陷）

判断顺序（命中即停）：

1. **明确说"bug / 缺陷 / 报错 / 异常 / 不符合预期"** → 缺陷
2. **输入里包含日志文件、堆栈、报错截图** → 缺陷
3. **明确说"需求 / 功能 / 优化 / 想加 / 希望支持"** → 需求
4. **描述的是"应该有什么 / 应该怎样"而当前完全没有** → 需求
5. **描述的是"现在是 X，但应该是 Y"且 Y 是已有功能的预期行为** → 缺陷
6. **以上都判断不出来** → 反问用户："这条想提需求还是缺陷？"

### 规则 2：必填项缺失才反问，可推断的不打扰

| 单类型 | 必填字段 |
|---|---|
| **需求** | 标题、描述（背景 / 详情 / 预期效果）、分类、优先级、提出人、项目组、处理人 |
| **缺陷** | 标题、处理人、严重程度、优先级、模块、描述（问题描述 / 重现步骤 / 实际效果 / 预期效果） |

- 能从上下文推断的：直接填，记录到"推断说明"
- 必填且无法推断的：反问，一次问完所有缺口（不要挤牙膏式追问）
- 选填字段：留空，不主动追问

字段映射与默认值规则见 `references/tapd_field_mapping.md`。

### 规则 3：日志文件按大小分档处理

| 日志大小 | 处理策略 |
|---|---|
| **< 200KB** | **B 模式（关键提取）**：扫描日志，挑出 ERROR / 异常堆栈 / 关键时间戳 / 业务关键字，放进"实际效果"段；原日志作为附件提示 |
| **≥ 200KB** | **A 模式（简单引用）**：只在描述里写"附日志 文件名，关键报错：[一行简要]"，不全文展开，避免污染上下文 |

判断流程：拿到日志后先 `ls -l` 或读取大小 → 决定走 A 还是 B；如果用户明确指定模式，按用户的来。

详细的日志关键信息提取规则见 `references/log_extract_rules.md`。

### 规则 4：描述段落按单类型用固定结构

**需求描述**（写到 description / 富文本字段）：

```
## 需求背景
[一句话说明为什么要做，关联的现状或业务场景]

## 需求详情
[功能要做成什么样，关键交互或规则，按需分点]

## 预期效果
[做完之后能解决什么问题 / 用户能感知到什么变化]
```

**缺陷描述**（写到 description / 富文本字段）：

```
## 问题描述
[一句话说明现象]

## 重现步骤
1. ...
2. ...
3. ...

## 实际效果
[当前实际表现，含报错信息 / 关键日志片段]

## 预期效果
[正确的应该是什么]
```

每个段落如果信息缺失，用 `[待补充]` 占位，不要硬编。

### 规则 5：标题写成"动作 + 对象 + 关键限定词"，10~20 字

不好的例子：「优化一下」「修一个 bug」「XX 项目需求」
好的例子：「列表批量操作按钮在筛选模式下失效」「文档列表支持按文件名模糊搜索」

## 工作流程

### Step 1：识别输入形态

判断输入是哪几类（可同时存在多种）：
- 文本（聊天 / 散记 / 会议纪要）
- 图片（聊天截图 / 报错截图）→ 需要 OCR 或视觉理解
- 日志文件 / 其他附件 → 走规则 3 的分档逻辑

### Step 2：判断单类型 + 提取核心信息

按规则 1 判断需求 / 缺陷，然后从输入里提取：
- 标题候选（按规则 5）
- 描述各段落的素材
- 任何能推断的字段（模块、优先级、提出人……）

### Step 3：补全必填字段

按规则 2 检查必填项，缺失的统一在一次回复里反问。

模板：

> 录入 TAPD 前，还需要你确认 N 项：
> 1. **[字段名]**：[为什么需要 / 候选值如果有]
> 2. ...
>
> 我先按以下推断填好了：
> - [字段]：[值]（依据：[来源]）

### Step 4：生成结构化结果 + 输出回执

输出两份内容：

**A. 给 TAPD MCP 用的结构化数据**（JSON 或字段表，看实际调用形态）

**B. 给用户看的回执**（用统一回执模板，见 `references/output_template.md`）

### Step 5：落单（如果用户确认）

调用 TAPD MCP（`stories_create` / `bugs_create`）写入。落单成功后回执里要带上 TAPD 链接。

## 统一回执模板

需求和缺陷共用同一份格式，差异只在前缀图标和字段名：

```
✅ 已提单 · [需求 / 缺陷]
─────────────────────────
📌 [标题]
🏷️ 优先级：P[X]  ·  分类/模块：[XX]  ·  处理人：[XX]
🔗 TAPD：[链接]

⚠️ 我做了 N 处推断，便于你核对：
  · [字段1]：[值]（依据：[来源]）
  · [字段2]：[值]（依据：[来源]）

📎 附件：[日志/截图，如有]
```

如果未落单（处于"等你确认"阶段），把首行换成：

```
🟡 待确认 · [需求 / 缺陷] 草稿
```

回执原则：
- **简洁优先**：不超过 8 行（不含推断列表）
- **推断必列**：所有 Skill 自己填的字段都要列出，让用户一眼能查
- **未填的选填项不列**：避免噪声

## 不做什么（边界）

- 不主动改 TAPD 已存在的单（除非用户明确说"更新某某单"）
- 不替用户做主观判断的字段（如"业务价值"「需求来源」这类需要表达立场的字段，留空让用户自己填）
- 不在用户没确认的情况下提交单（默认走"草稿 → 用户确认 → 提交"两步式）

## 与其他 Skill 的协作

- 需要从聊天/会议提取多条需求时：本 Skill 一次只处理一条；如有多条，先列清单让用户确认，再逐条走流程
- 涉及竞品调研、PRD 撰写：交给 `prd-context-fetch` 和 `prd-writer`
- 单子提完后想沉淀复盘：交给 `case-distill`
