# TAPD 字段映射表与默认值规则

> 给 `tapd-requirement-record` Skill 用。每次落单前先查一遍这份表，按"能推断就填，必填项缺失才反问"的原则操作。

## 一、需求单字段（必填）

| 字段 | TAPD 字段名 | 推断规则 / 默认值 | 缺失策略 |
|---|---|---|---|
| 标题 | `name` | 从输入里抽核心动作+对象+限定词，10~20 字；不啰嗦 | 必反问 |
| 描述 | `description` | 按"需求背景 / 需求详情 / 预期效果"三段式渲染富文本；每段缺信息用 `[待补充]` 占位 | 三段都为空才反问 |
| 分类 | `category_id` | 看输入提到的模块词（按下方"模块词典"映射）直接对应；没明说时根据上下文推断 | 实在拿不准才反问 |
| 优先级 | `priority` | 默认 `Middle`；如果输入里有"紧急 / 阻塞 / 立刻"→ `High`；"以后再说 / 优化下"→ `Low` | 不反问，按默认+推断 |
| 提出人 | `creator` | 优先用聊天里的发言人；没有则填当前用户 `{姓名}` | 反问"提出人是谁" |
| 项目组 | `iteration_id` 或自定义字段 | 默认填当前所在项目（`<YOUR_PROJECT>` 主线）；用户提到具体迭代时按用户的 | 必反问 |
| 处理人 | `current_owner` | 默认填 `{姓名}`（当前用户）；用户在输入里点名了别人就覆盖（详见第四节） | 不反问 |

## 二、缺陷单字段（必填）

| 字段 | TAPD 字段名 | 推断规则 / 默认值 | 缺失策略 |
|---|---|---|---|
| 标题 | `title` | 从报错现象里抽，结构："[模块] [操作] [现象]"，例如"列表 批量编辑模式下 取消选区按钮失效" | 必反问 |
| 处理人 | `current_owner` | 默认填 `{姓名}`；用户点名了别人就覆盖（详见第四节） | 不反问 |
| 严重程度 | `severity` | 默认 `Normal`；功能完全不可用→`Serious`；阻塞主流程→`Fatal`；体验小问题→`Prompt` | 不反问 |
| 优先级 | `priority` | 默认 `Middle`；阻塞用户使用→`High`；偶现/可绕过→`Low` | 不反问 |
| 模块 | `module` | 按下方"模块词典"推 | 必反问 |
| 描述 | `description` | 按"问题描述 / 重现步骤 / 实际效果 / 预期效果"四段式 | 四段都为空才反问 |

## 四、处理人默认规则

> 不同团队对"处理人"的分派策略不同。这里给出两类常见策略，根据团队实际情况二选一。

### 策略 A：临时分派型团队（推荐默认）

适用：模块研发对接人不固定、按需求/缺陷临时分派的团队。

- **不维护"模块→人"的硬映射表**（容易过期）
- **默认规则**：处理人一律默认填 `{姓名}`（即当前用户自己），让所有单子先挂在自己名下，避免漏
- **覆盖条件**：当且仅当用户在输入里**明确点名**了具体的人（例如"这个让小王跟一下"、"派给 @张三"、"XXX 你看下这个 bug"），才把处理人覆盖为该人名；否则保持 `{姓名}` 不变，**不要反问**
- **回执提示**：当处理人是 `{姓名}`（默认值）时，在回执的"⚠️ 我做了 N 处推断"里加一条："处理人默认填了你自己（`{姓名}`），如果要派给别人，告诉我名字我改"，让用户有覆盖入口

### 策略 B：固定分派型团队

适用：模块和研发负责人有稳定 1:1 映射的团队。

在本目录加一份 `module_owner_mapping.md`，维护"模块→处理人"映射；推断时优先查这份表，未命中再回退到 `{姓名}`。

### 示例（按策略 A）

- 输入"{模块 A} 抓不到数据，建个 bug" → `current_owner = {姓名}`（无人名，走默认）
- 输入"{模块 A} 抓不到数据，建个 bug，让老李看" → `current_owner = 老李`（命中点名，覆盖）
- 输入"{模块 B} 模式按钮失效" → `current_owner = {姓名}`（无人名，走默认）

## 五、选填字段（默认不填，不主动追问）

- 业务价值、需求来源、商业化收益等需要主观立场的字段
- 计划开始/结束时间
- 预估工作量
- 关联需求 / 关联缺陷
- 标签 tag

如果用户在输入里主动提到了这些，按用户原话填；没提就留空。

## 六、提交前自检 checklist

落单前 Skill 自己跑一遍：

- [ ] 必填字段是否全部有值（含推断值）？
- [ ] 描述段落是否都有内容（不是空的）？
- [ ] 标题长度是否在 10~20 字、是否包含动作和对象？
- [ ] 推断字段是否都列在回执的"⚠️ 我做了 N 处推断"里？
- [ ] 如果有日志附件，是否按 `log_extract_rules.md` 处理过？

## 配置说明

把这份 Skill 接入到自己的项目时，必须改的地方：

1. **第一节 / 第二节**里的 `{姓名}` → 改成你自己的姓名或团队约定的处理人默认值
2. **第一节 / 第二节**里的 `<YOUR_PROJECT>` → 改成你自己的项目名
3. **第三节"模块词典"**：整张表都要按你自己项目的模块结构改写——这是最重要的本地化项
4. **第四节"处理人默认规则"**：根据团队实际选择策略 A 或策略 B
5. **第二节缺陷标题示例**里的"列表 批量编辑模式下 取消选区按钮失效"可以替换成更贴近你自己项目的示例
