# 版本变更记录表 —— 模板与填写规范

## 模板（直接复制到 PRD 开头）

```markdown
## 版本变更记录 <!-- anchor: changelog -->

| 版本 | 更新时间 | 改动摘要 | 影响范围 | 修改人 |
|------|----------|----------|----------|--------|
| v0.1 | YYYY-MM-DD HH:MM | 初稿 | 全文 | {姓名} |
```

## 字段填写规范

### 版本号

| 类型 | 何时使用 | 示例 |
|------|----------|------|
| `v0.x` | 内部讨论稿，未评审 | v0.1, v0.2, v0.3 |
| `v1.0` | 首次评审通过、进入开发的基线 | v1.0 |
| `v1.x` | 基线后的小改：细节调整、边界补充、文案微调 | v1.1, v1.2 |
| `v2.0+` | 架构级或核心流程级变更 | v2.0, v3.0 |

### 更新时间

- 格式：`YYYY-MM-DD HH:MM`（精确到分钟）
- 用修改当时的实际时间，不用预计上线日期
- 同一天可能改多次，时间戳能区分先后顺序，便于追查"研发拿到的是哪一版"

### 改动摘要

**写作三条铁律**：

1. **动词开头**：新增 / 删除 / 修改 / 调整 / 拆分 / 合并 / 重写
2. **一行一件事**：超过 3 件事就拆多行（在单元格内用 `<br>` 换行）
3. **必带锚点引用**：用 `feat-X-Y` 格式指向具体章节

#### 示例对比

✅ 好的写法：
- `修改 feat-2-1 项目列表筛选项，新增"按状态"选项`
- `新增 feat-3-3 批量删除二次确认弹窗`
- `调整 feat-1-1 模糊搜索的边界，明确不区分大小写`

❌ 不好的写法：
- `调整了一些筛选逻辑` （太模糊，没锚点）
- `优化用户体验` （没说改了什么）
- `根据评审意见修改` （没说具体改了哪里）

### 影响范围

填法：列出受影响的章节锚点；如有同步改原型，加 `+原型 prototype/xxx.html`

#### 示例

| 影响范围（这一列怎么填） | 何时这样填 |
|--------------------------|-----------|
| `全文` | 初稿、整体重写 |
| `feat-2-1` | 只改一个章节 |
| `feat-2-1, feat-3-3` | 改多个不相邻章节 |
| `feat-2-1 +原型 prototype/project-list.html` | 改章节同时同步改了对应 HTML 原型 |
| `feat-2 整章` | 改整个 1 级模块 |

### 修改人

- 一般填 `{姓名}`
- 如果是基于他人反馈改的，可以写 `{姓名}（基于研发评审）`、`{姓名}（基于设计反馈）` 等
- 如果是从原型反向同步回 PRD（D 分支），写 `{姓名}（基于原型反推）`

## 完整填写示例

```markdown
## 版本变更记录 <!-- anchor: changelog -->

| 版本 | 更新时间 | 改动摘要 | 影响范围 | 修改人 |
|------|----------|----------|----------|--------|
| v0.1 | 2026-05-10 14:20 | 初稿 | 全文 | {姓名} |
| v0.2 | 2026-05-12 10:35 | 新增 feat-3 批量操作模块<br>调整 feat-2-1 筛选项顺序 | feat-2-1, feat-3 | {姓名} |
| v1.0 | 2026-05-15 16:00 | 评审通过，无改动 | 全文 | {姓名} |
| v1.1 | 2026-05-18 09:50 | 修改 feat-2-1 增加"按状态"筛选选项 | feat-2-1 +原型 prototype/project-list.html | {姓名}（基于研发评审） |
| v1.2 | 2026-05-19 23:55 | 新增 feat-4-2 空状态文案<br>修改 feat-3-3 删除确认文案 | feat-3-3, feat-4-2 | {姓名} |
| v1.3 | 2026-05-20 11:10 | 新增 feat-2-1 排序按钮（原型上先加的） | feat-2-1 +原型 prototype/project-list.html | {姓名}（基于原型反推） |
```

## 何时升级版本号（速查）

| 改动类型 | 版本号变化 |
|----------|-----------|
| 撰写中改细节、补边界 | v0.1 → v0.2 → v0.3 |
| 评审通过 | v0.x → v1.0 |
| 基线后改文案、补边界、调小细节 | v1.0 → v1.1 → v1.2 |
| 基线后大改流程/架构 | v1.x → v2.0 |
