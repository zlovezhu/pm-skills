# PRD Writer —— 产品需求文档写作助手

## 概述

帮助产品经理快速撰写、维护结构化的产品需求文档（PRD）。本 Skill 在写作能力之外，还承担两件事：

1. **版本可追溯**：PRD 每次改动都留痕，研发能一眼看出"这版和上版差什么"
2. **可选生成 HTML 低保真原型**：仅在用户明说时触发，不默认生成

## 适用场景

- 从零撰写一个功能模块的完整需求稿
- 修改/扩充已有 PRD（**强制走版本变更流程**）
- 根据原型图/截图补充需求细节
- 将口头描述转化为规范化需求文档
- 用户明说要"HTML 原型 / 可点击 demo / 画个原型"时，附带生成低保真 HTML

## 触发分支判断

进入 Skill 时先判断走哪条分支：

| 用户表达 | 分支 |
|---|---|
| 「写个需求」「起一份 PRD」「这个功能写成需求稿」 | **A. 新建 PRD** |
| 「改一下 X 章节」「补充一下 Y 的边界」「这版加上 Z」 | **B. 修改已有 PRD** |
| 上面任意一条 + 「帮我生成 HTML 原型」「画个原型」「出个可点击 demo」 | **附加 C. 生成 HTML 原型** |
| 「我改了原型」「原型上加了 XX」「把原型改动同步回 PRD」 | **D. 原型反向同步到 PRD** |

**关键规则**：
- 分支 A / B 在动笔前，**复杂需求必须先走"需求拆解讨论"环节**（详见 §〇 写作前置）
- 分支 B 必须先读旧版本表，再追加变更行（详见下文「版本管理」）
- 分支 C 是可选叠加项，**用户不明说就不做**；不要主动建议、不要"顺手生成"
- 分支 D 触发后必须先做 diff 判断，**只有"产品决策类"差异才回流 PRD**（详见 §3.6）

## 〇、写作前置：需求拆解讨论（A / B 分支）

不允许直接动笔写 PRD。复杂需求必须先和用户对齐"考虑点清单"，避免写完反复改。

### 0.1 是否触发拆解讨论

按以下规则判断，**简单需求直接跳过、复杂需求强制走**：

| 场景 | 是否触发 |
|---|---|
| 简单需求：单点改动、文案调整、加个字段、改个交互 | ❌ 跳过，直接写 |
| 复杂需求（命中下方任一硬指标） | ✅ 强制走，不能跳 |
| 用户主动说"先讨论再写""帮我先拆一下" | ✅ 强制走 |
| 用户主动说"直接写吧""不用讨论" | ❌ 跳过 |

**复杂判断硬指标**——从「性质 / 深度 / 广度」三个维度看，命中任一即视为复杂：

#### 维度 1：改动性质（最强信号，优先级最高）

不论广度多小，命中任一即视为复杂：
- 新增 / 删除流程分支或状态机节点
- 改变核心交互路径（用户操作步骤变了）
- 新增 / 修改字段含义、数据结构、权限规则
- 涉及上下游接口契约变更
- 涉及历史数据兼容、迁移、回滚
- 涉及权限、计费、外部依赖
- 涉及异步任务、并发、跨端同步
- 涉及 ≥ 2 个模块 / 角色

#### 维度 2：改动深度（单章节内深改也算）

哪怕只改 1 个章节，命中任一即视为复杂：
- 单个 1/2 级章节内 ≥ 3 个子条目被改 / 删 / 加
- 单个章节整体重写（不是局部调整）
- 改了该章节的"核心逻辑"而非"边界细节"

#### 维度 3：改动广度

- 跨 ≥ 3 个 1/2 级章节
- 或跨 ≥ 2 个章节 **且** 任一章节命中维度 2

#### 输入复杂度兜底（适用于 A 分支）

- 用户输入的需求描述 ≥ 200 字 或包含多个 "并且 / 同时 / 还要"

### 0.1.1 反向兜底：明确不触发拆解的"简单改动"

为了避免过度触发，以下场景**直接跳过拆解**（即使凑巧命中了上面的硬指标，也以本节为准）：

- 纯文案 / 提示语 / 报错信息修改
- 调整字段顺序、显隐、默认值（不改含义）
- 修正笔误、补充示例、调整排版
- 加 `> 💬` 拆解记录提示本身
- 只改延期标记（灰色字体 / "（暂时不上）" 标注）
- 只改章节标题（不动正文条目）

### 0.1.2 安全阀：拿不准时反问一句

复杂判断本质是启发式，边界 case 难免存在。AI 拿不准是否要走拆解时，**直接反问用户一句**：

> 「这次改动我倾向走拆解讨论，因为 {具体原因，例如"涉及状态机新增" / "改了 3 个章节"}，OK 吗？」

- 用户回「不用，直接写」→ 跳过拆解
- 用户回「OK / 走一下」→ 进入 §0 拆解流程
- 用户没明确表态 → 按"复杂"处理，宁愿多对齐一次

### 0.2 拆解维度清单

按下面十个维度组织讨论，**不强制每条都列**——根据本次需求实际涉及的范围按需选取，无关维度直接省略，避免清单注水：

1. **目标与边界**：这次解决什么、不解决什么
2. **用户与场景**：谁用、什么情况下用、频率
3. **核心流程**：主路径 + 关键分支
4. **数据与字段**：新增/修改哪些字段、数据来源、保留策略
5. **状态与异常**：状态机、异常分支、超时/失败/回滚
6. **权限与安全**：谁能看、谁能改、敏感数据
7. **性能与容量**：数据量级、超时阈值、并发预期
8. **依赖与改造**：依赖哪些上下游、是否需要他们配合
9. **埋点与度量**：怎么验证有没有用、关键指标
10. **验收标准**：上线判定条件

> 团队特殊关注点（如灰度、AB、降级方案）若本次需求涉及，可挂在"目标与边界"或"依赖与改造"里，不另起新维度。

### 0.3 输出形态：一次性清单

把判定为"涉及"的维度一次性列出，**不做一问一答**。每条按以下结构：

```
N. {维度名}
   ✅ 已推断：{从用户描述/截图/历史素材里能确定的内容}
   ❓ 待确认：{需要用户回答的问题，编号 Q1/Q2/...}
   💡 建议：{可选，我倾向的方案 + 理由}
```

**约束**：
- 已推断部分要具体，不要"已了解需求背景"这种空话
- ❓ 待确认问题用 `Q1/Q2/...` 编号，方便用户挑着回
- 💡 建议必须给"理由 + 倾向选项"，不要只罗列选项

### 0.4 收敛规则

- 用户针对 ❓ 编号回复即可，没回复的视为认可"已推断"和"💡 建议"
- 用户全部对齐后说「OK 开始写 / 可以了 / 继续」，才进入正文撰写
- 拆解过程中如果用户改了想法，更新清单再确认一次，不要直接带着旧理解动笔

### 0.5 PRD 内留痕（轻量）

拆解讨论结论**不单独成章**，而是直接挂在对应需求点旁边作为提示，格式：

```markdown
### 2.1 批量删除 <!-- anchor: feat-2-1 -->

> 💬 拆解记录：单次最多 50 条；删除后软删 30 天；不支持跨项目批量。

2.1.1 批量删除入口：...
```

约束：
- 一行写完，超过两行说明这条结论太重，应该写进正文需求条目而不是挂提示
- 只挂"讨论中达成的、容易遗忘的边界",已经写进正文的不重复挂
- B 分支修改时，原有 `> 💬` 提示不动，除非本次讨论改了那条结论

## 一、写作规范（A、B 共用）

### 1.1 编号层级结构

采用 `X.Y.Z.W` 四级编号体系：

```
第1级（X）    ：功能模块名称（加粗），如「1 搜索：」「2 角色筛选：」
第2级（X.Y）  ：模块下的主要功能点，如「3.3 手动添加：」
第3级（X.Y.Z） ：功能点的子项/组件，如「3.3.1 成员输入框：」
第4级（X.Y.Z.W）：具体的交互规则和细节
```

### 1.2 章节稳定锚点

**每个第 1 级和第 2 级标题都必须挂一个稳定锚点**，便于变更摘要引用、便于 HTML 原型反查映射：

```markdown
## 1 搜索 <!-- anchor: feat-1 -->
### 1.1 模糊搜索 <!-- anchor: feat-1-1 -->
```

锚点命名规则：
- 第 1 级：`feat-{编号}`
- 第 2 级：`feat-{编号}-{编号}`
- **锚点一旦建立，后续修改章节内容时不要变更 ID**（即使章节编号变了，锚点也保留原值，避免变更追踪失效）

### 1.3 格式规则

- **第1级标题**：使用加粗字体，以冒号结尾，如 **「1 搜索：」**
- **第2级小标题**：需要展开描述的子功能，加粗或半粗，以冒号结尾
- **第3级/第4级**：普通文本，描述具体的交互规则和细节
- **暂不上线的功能**：使用灰色字体（`#C7C7CC`）标注，并加括号说明「（暂时不上）」
- **原型图位置**：在关键功能点描述后预留原型图截图位置

### 1.4 内容写作要求

每条需求必须包含：

1. **具体的用户操作**：「点击」「输入」「选择」「拖拽」等动作
2. **操作的对象**：按钮名称、输入框、下拉框等 UI 元素
3. **操作的结果**：弹出弹窗、刷新列表、显示提示等反馈
4. **边界情况处理**：空状态、必填校验、错误提示等

写作语言风格：

- 使用**陈述句**，直接描述功能行为
- 以功能为主语（非用户），如「支持模糊搜索」而非「用户可以模糊搜索」
- 具体化所有 UI 元素名称，用【】标注按钮名称，如【取消】【确认添加】
- 涉及格式时给出具体示例，如「格式：YYYY-MM-DD（如：2022-03-15）」
- 涉及数量或文案时给出精确内容，如「显示"已选择 X人"」

### 1.5 典型需求条目模式

参见 `references/prd_example.md`，该文件包含搜索/筛选/弹窗/列表/批量操作/删除确认/空状态/分页等常见模式。

## 二、版本管理（B 分支强制走，A 分支也要建初始版本表）

### 2.1 版本表必须出现在 PRD 开头

每份 PRD 文件**第一个 H2 章节就是版本表**，位置固定，模板见 `references/version_table_template.md`：

```markdown
## 版本变更记录 <!-- anchor: changelog -->

| 版本 | 更新时间 | 改动摘要 | 影响范围 | 修改人 |
|------|----------|----------|----------|--------|
| v0.1 | YYYY-MM-DD HH:MM | 初稿 | 全文 | {作者名} |
```

**字段说明**：
- 「更新时间」精确到分钟，格式 `YYYY-MM-DD HH:MM`，方便同一天多次修改时区分先后
- 「修改人」必填，多人协作时一眼看出谁改的

### 2.2 版本号约定

- `v0.x`：内部讨论稿（撰写中、未评审）
- `v1.0`：评审通过、可进入开发的基线版本
- `v1.1 / v1.2 / ...`：基线后的小改（细节调整、边界补充）
- `v2.0 / v3.0 / ...`：架构级或核心流程变更后的大版本

### 2.3 修改 PRD 的强制流程（B 分支必须遵守）

不允许 AI 直接改正文不留痕。每次修改必须按以下顺序：

1. **读旧版本表**：找到表格最末一行，确定当前最新版本号
2. **生成变更摘要**：
   - 列出本次具体改动（哪些章节增/删/改、什么改动）
   - 用锚点引用涉及章节，如 `feat-2-1 增加批量删除确认`
3. **和用户确认变更摘要**：
   - 输出待写入的变更行 + 准备改的正文 diff 概要
   - **必须等用户回"OK / 继续 / 改一下 XX"** 才动正文
4. **写入**：
   - 在版本表末尾追加新行（更新时间填**当前时刻**，精确到分钟）
   - 改正文
   - 如果同时改了 HTML 原型（C 分支），版本表"影响范围"列要标注「+原型 prototype/xxx.html」

### 2.4 变更摘要写作要求

- 使用动词开头：新增 / 删除 / 修改 / 调整 / 拆分 / 合并
- 一行一件事，超过 3 件事就分多行
- **必须带锚点引用**，例如：
  - ✅ `修改 feat-2-1 项目列表筛选项，新增"按状态"选项`
  - ❌ `调整了一些筛选逻辑`（太模糊）

## 三、HTML 低保真原型（C 分支，可选）

### 3.1 触发条件（严格）

只有用户**明说**才生成。明说的判断标准：

- ✅ 触发：「帮我生成 HTML 原型」「画个原型」「出个可点击 demo」「同步生成 HTML」「画一下页面」
- ❌ 不触发：「这个功能要有个列表页」（这是需求描述，不是要原型）
- ❌ 不主动建议：写完 PRD 后**不要**问「需要我生成 HTML 原型吗」，除非用户主动问起

### 3.2 覆盖范围

- **只覆盖核心流程页**：3~5 个，最多不超过 5 个
- 不要每个二级章节都画
- 由 Skill 主动判断哪些算"核心流程"：通常是用户主路径上的页面（列表页、关键弹窗、详情页），不画规则说明类的内容
- 输出前先和用户确认要画哪几个页面，**用户回 OK 再开始生成**

### 3.3 文件组织

```
{PRD 文件名}.md
prototype/
├── index.html          # 导航页，按 PRD 章节顺序串起所有原型页
├── {页面英文名}.html    # 例如 project-list.html
├── {页面英文名}.html
└── styles.css           # 共享样式
```

### 3.4 HTML 编写规范

- **极简风格**：灰白配色、Tailwind 默认样式或简单的自写 CSS，不追求好看
- **统一 class 命名**：
  - `.pf-page`：页面容器
  - `.pf-card`：卡片块
  - `.pf-table`：列表表格
  - `.pf-button-primary` / `.pf-button-secondary`：主/次按钮
  - `.pf-modal`：弹窗
  - `.pf-empty`：空状态
- **每个 HTML 顶部加 PRD 锚点注释**：

```html
<!--
  @prd: feat-2-1 项目列表页
  @prd-file: prd-project-list.md
  @last-sync: v1.2 / YYYY-MM-DD
-->
```

- **不要写花哨的动画/渐变/阴影**，留给设计同学在 Figma 阶段做

### 3.5 同步策略

**默认方向：PRD → 原型（正向）**
- PRD 是 source of truth
- 修改 PRD 章节时，如果该章节有对应原型页（看锚点注释），**必须同步改对应 HTML**
- 改完后在版本表"影响范围"列标注 `+原型 prototype/project-list.html`
- HTML 顶部 `@last-sync` 注释也要更新到最新版本号

**反向方向：原型 → PRD（D 分支，需 diff 判断）**
- 用户在原型上调整后，可触发 D 分支把改动回流 PRD
- **不是无脑回流**，先做 diff 判断（详见 §3.6）

### 3.6 D 分支：原型反向同步到 PRD

#### 触发条件
用户明说「我改了原型」「原型上加了 XX」「把原型改动同步回 PRD」之类。

#### 执行步骤

1. **读取对比**：
   - 读 HTML 原型当前内容（含 `@prd` 锚点注释）
   - 读 PRD 中该锚点对应的章节
2. **生成 diff 清单**，按以下分类：

| 类别 | 是否回流 PRD | 举例 |
|------|--------------|------|
| **产品决策类** | ✅ 回流 | 新增按钮/新增交互状态/新发现的边界情况/新增字段/状态文案变更 |
| **PRD 已写但原型未实现** | ⚠️ 反提醒，不回流 | 原型缺了 PRD 已定义的功能，提示用户"是不是原型漏画了" |
| **纯视觉调整** | ❌ 不回流 | 颜色/间距/字号/对齐方式/动画效果 |
| **临时占位** | ❌ 不回流 | 测试数据、Lorem ipsum、demo 用的假状态 |

3. **输出判断结果**：
   - 如果 diff 清单里**没有产品决策类**差异 → 直接告知用户「原型和 PRD 在产品维度一致，无需回流」，**不动 PRD**
   - 如果有产品决策类差异 → 列出清单，每条标注"建议回流到 feat-X-Y"，等用户确认
4. **用户确认后走 B 分支标准流程**：
   - 追加版本行（"修改人"列建议写 `{作者名}（基于原型反推）`）
   - 改正文
   - HTML 顶部 `@last-sync` 也更新

#### 关键约束
- **判断要保守**：拿不准是不是产品决策类时，列出来让用户判断，不要替用户做主
- **批量修改场景**：如果 diff 出来一堆，先按章节分组展示，不要一次性塞给用户

### 3.7 不做的事

- ❌ 不做 HTML → Figma 自动转换（Figma 设计稿由设计同学接手）
- ❌ 不做高保真原型
- ❌ 不在 PRD 末尾自动加"如需 Figma 设计稿，请..."这种废话提示

## 四、工作流程

### 分支 A：新建 PRD

1. **理解需求**：功能名称、所属页面、涉及角色、有无截图参考
2. **判断是否复杂**（按 §0.1 三维度判断 + §0.1.1 反向兜底 + §0.1.2 安全阀）：复杂则进入 §0 拆解讨论环节，等用户对齐再继续
3. **拆分模块**：拆为顶级模块（搜索/筛选/新增/列表/编辑/删除/批量/分页/空状态）
4. **建版本表**：写入 v0.1 初稿行
5. **逐模块撰写**：按编号层级体系展开，每个 1/2 级标题挂锚点；拆解讨论的边界结论挂 `> 💬` 提示
6. **补充边界情况**：必填校验、空状态、权限、成功/失败反馈、刷新时机
7. **标注延期功能**：灰色字体 + 「（暂时不上）」
8. **如果用户明说要 HTML 原型**：进入 C 分支流程

### 分支 B：修改已有 PRD

1. **读 PRD 文件**：找到版本表，确定当前最新版本号
2. **理解修改点**：用户要改哪几个章节、改成什么样
3. **判断是否复杂**（按 §0.1 三维度判断 + §0.1.1 反向兜底 + §0.1.2 安全阀）：复杂则进入 §0 拆解讨论环节
4. **输出变更预览**：
   - 准备追加的版本行（版本号 + 日期 + 改动摘要 + 影响范围）
   - 准备改的正文 diff 概要（锚点 + 改前/改后简述）
5. **等用户确认**（必须，不能跳过）
6. **执行修改**：
   - 追加版本行
   - 改正文（涉及的章节锚点保持不变；新讨论结论可补 `> 💬` 提示）
   - 如有对应 HTML 原型，同步改并更新顶部 `@last-sync`

### 分支 C：生成 HTML 原型（叠加）

1. **判断核心流程页**：从 PRD 中选 3~5 个用户主路径页面
2. **和用户确认要画的页面清单**
3. **生成 HTML**：
   - 每页一个文件，统一 class 命名
   - 顶部加 `@prd` 锚点注释
   - 写一个 `index.html` 导航页串起来
   - 写一个 `styles.css` 共享样式
4. **更新 PRD 版本表**：在影响范围列加 `+原型 prototype/xxx.html`

### 分支 D：原型反向同步到 PRD

1. **读取对比**：HTML 当前内容 vs PRD 对应章节（按 `@prd` 锚点定位）
2. **生成 diff 清单**：按 §3.6 表格分类（产品决策类/PRD 已写未实现/纯视觉/临时占位）
3. **判断是否需要回流**：
   - 没有产品决策类差异 → 告知用户"无需回流"，结束
   - 有产品决策类差异 → 列清单，等用户确认
4. **用户确认后走 B 分支标准流程**：追加版本行、改正文、更新 HTML 顶部 `@last-sync`
