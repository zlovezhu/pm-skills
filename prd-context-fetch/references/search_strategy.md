# 分层检索策略

> 给 `prd-context-fetch` Skill 用。这份文档描述如何在 Obsidian 和 TAPD 里跑检索、如何排序、什么时候走兜底。

## 一、检索源与默认开关

| 源 | 默认状态 | 触发条件 | 排除条件 |
|---|---|---|---|
| Obsidian `wiki/` | ✅ 开 | 始终检索 | 无 |
| Obsidian `card/` | ✅ 开 | 始终检索 | 无 |
| Obsidian `raw/` | ❌ 关 | 仅当 wiki + card 全空 | wiki 或 card 有命中时 |
| TAPD 当前项目同模块需求 | ✅ 开 | 始终检索 | 无 |
| TAPD 跨项目 / 已拒绝单 | ❌ 关 | 用户明确说"也看一下其他项目" | 默认不查 |

## 二、Obsidian 检索的具体跑法

### 2.1 拿当前 tag 列表

调 Obsidian MCP（`search_vault` 或对应 list_tags 接口），拿全部 tag。

把用户输入的"检索目标"做关键词归一化：
- "{模块名} {子特性}" → 归一化 token: ["{模块名}", "{子特性}"]
- "{模块名} 加载慢" → ["{模块名}", "性能"]

和 tag 列表做：
1. **精确匹配**：tag 名直接包含 token → 强匹配
2. **语义近似**：tag 名和 token 语义相近（如 token = "性能"，tag = "perf-tuning"）→ 弱匹配

输出 top 3 候选 tag，强匹配排前面。

### 2.2 跑全文检索

对每个选中的 tag，分别在 `wiki/` 和 `card/` 目录下跑：

```
search_vault(
  query = 检索目标的核心词,
  tag   = <选中 tag>,
  path  = "wiki/" 或 "card/"
)
```

### 2.3 排序规则（相关度）

按下面四个维度打分（满分 10 分），加权求和：

| 维度 | 权重 | 说明 |
|---|---|---|
| 标题命中关键词数量 | 4 | 命中 1 个 = 2 分，2 个 = 4 分（封顶） |
| 正文命中关键词频次 | 3 | log(频次+1)，最高 3 分 |
| 笔记新鲜度 | 2 | 30 天内 = 2 分，半年内 = 1 分，更早 = 0 分 |
| Tag 强匹配 | 1 | 强匹配 = 1 分，弱匹配 = 0.5 分 |

按总分倒序，wiki 取 top 5，card 取 top 3。

## 三、TAPD 检索的具体跑法

### 3.1 检索条件

```
project_id  = 当前项目（默认 <YOUR_PROJECT>，可由用户切换）
module      = 用户在 Step 2 勾选的模块
status      ∈ ['已发布', '已关闭', '开发中', '待发布']
modified    > now - 6 months
order by    = modified desc
limit       = 10
```

### 3.2 输出字段

每条历史需求只取：
- TAPD ID
- 标题
- 模块
- 状态
- 最近修改时间
- 描述前 200 字（用作"关键点摘要"基础素材）

### 3.3 关键点提取

不把整份描述塞回 context，要从描述里抽 1~3 条"关键点"：
- 已知约束（如"不支持 iOS 系统版本 < 14"）
- 边界情况（如"用户未授权 IDA 时降级"）
- 设计决策（如"采用异步加载，首屏不阻塞"）

抽取规则：扫描描述里包含"约束 / 限制 / 注意 / 边界 / 异常 / 兜底 / 不支持 / 默认"等关键词的句子，取首 3 条。

## 四、兜底降级的判定

走 raw 兜底的严格条件：

```
wiki 命中数 == 0 AND card 命中数 == 0
```

（TAPD 命中不影响降级判断——TAPD 是平行通道）

降级时：
- 在输出最顶部加一段警告：

  > ⚠️ 当前模块在 wiki/ 和 card/ 没有命中任何沉淀，已降级到 raw/ 检索。raw 是未消化的原始素材，仅供参考，不应直接作为 PRD 依据。

- raw/ 取 top 3，排序规则同 2.3
- 在每条 raw 结果后追加："建议本次写完 PRD 后用 `case-distill` 把这块沉淀成 wiki"

## 五、输出截断与上下文控制

为避免 PRD 撰写时 context 过长：

| 项 | 截断规则 |
|---|---|
| wiki 摘要 | 每篇 200 字内 |
| wiki 关键段落原文 | 每篇 500 字内，多段落用 `---` 分隔 |
| card 内容 | 全文（card 本就短，一般 < 200 字） |
| TAPD 关键点 | 每条需求最多 3 条 |
| 总 context 大小 | 控制在 8000 token 内；超过则按相关度倒序丢弃 |

## 六、自检 checklist

返回给用户前 Skill 自跑：

- [ ] 是否调过 Obsidian MCP 拿过当前 tag 列表（不是用了硬编 tag）？
- [ ] wiki + card 是否都跑过？
- [ ] TAPD 检索是否限定在当前项目 + 同模块？
- [ ] 如果走了 raw 兜底，是否加了警告横幅？
- [ ] 每条结果是否都有"相关度依据"那一句话？
- [ ] 默认勾选状态是否合理（强相关勾、弱相关不勾）？
